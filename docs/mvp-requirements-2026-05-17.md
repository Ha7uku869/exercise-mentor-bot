# 筋トレメタ認知MVP 要件定義・友人フィードバック整理

date: 2026-05-17
tags: #portfolio #mvp #fitness-ai #requirements

## 1. プロダクト概要

筋トレや運動の記録を、チャット入力から自動で構造化し、表・グラフで可視化する運動管理MVP。

ユーザーは「今日ベンチプレス60kgを8回3セットやった」「30分ランニングして5km走った」のように自然文で入力できる。  
LLM Function Calling により運動記録を抽出し、Supabase に保存する。  
また、怪我・目標・生活変化などの文脈情報も context_notes として保存し、次回以降の応答に活用する。

## 2. 作成背景

筋トレ中に「前回どの重量でやったか」「最近どの部位をどれくらいやったか」を毎回覚えたり、メモを探したりするのが面倒だった。  
そこで、過去記録や現在の状態をチャット・表・グラフで包括的に可視化できる身近な運動管理システムがあれば便利だと考えた。

## 3. 現在実装済みの主な機能

- WebチャットUI
- ニックネーム + 合言葉による簡易ユーザー分離
- チャット自然文から運動記録を抽出
- workouts テーブルへの保存
- context_notes テーブルへの怪我・目標などの保存
- 種目別グラフ可視化
- ランニング・ヨガ・筋トレなど複数運動タイプへの対応
- chat_logs にユーザー入力・AI返答・保存結果を記録
- Vercel + Render + Supabase による公開

## 4. 友人に依頼した検証観点

- 使ってみて分かりにくいところがあるか
- 筋トレ記録を管理する上で便利そうか
- 逆に不要に感じる機能があるか
- 追加されたら嬉しい機能があるか
- 画面や操作感で改善した方がいいところがあるか

## 5. 友人フィードバック整理

### 5.1 入力・記録修正系

- 種目名の訂正ができるようにしてほしい
- チャット入力だけだと面倒に感じる人もいそう
- 定型入力フォームやセレクトメニューもあるとよさそう
- 回数・時間・距離などはフォームで入れられると分かりやすいかもしれない

### 5.2 会話体験・UI系

- 一度会話を終えて再度開いたとき、前回の会話ログが残らない
- 「ベンチプレスの記録がありません」という文章が残り続けて違和感がある
- 初期状態で何ができるかが分かりづらい
- プレースホルダーや例文で入力例を示すとよさそう
- エラーコード500をそのままユーザーに見せる必要はなさそう

### 5.3 アカウント・認証・セキュリティ系

- 新規登録とログインの違いが分かりづらい
- AのアカウントにBがログインしようとした場合の挙動が曖昧
- user_id が localStorage に保存され、クエリパラメータで GET に使われている点が気になる
- ログイン状態やセッション管理が弱い
- 別端末からログインされた場合に検知できるか不明
- AIチャットを公開するなら利用量制限が必要
- 目的外利用を防ぐ仕組みが必要
- 本番化するならメール認証や正式なAuthが必要

## 6. 要件定義

### 6.1 直近MVPで優先して改善する要件

#### 要件A: 初期状態で何ができるか分かるようにする(? ヘルプタブ方式)

目的:
初めて使う友人が、「このアプリで何ができるか」をワンクリックで確認できるようにする。

採用方針(2026-05-17 最終):
当初の placeholder 改善案 → チュートリアル画面案 を経て、最終的に **画面内の「?」ヘルプタブ** で対応する。

スコープ縮小の理由:
- 火曜(2026-05-19)のトヨタES締切を考えると、チュートリアル画面実装は工数過剰
- 「実際の使用イメージ」は **面接スライド / README に載せる方が伝わる**(面接官は MVP を触らない前提)
- アプリ画面内には「迷ったときの最小限の助け舟」だけあればよい
- 既存の placeholder (ログイン画面の `例: haruku, alice, りんご` / チャット欄の操作説明) は役割が違うため変更しない

対応案:
- ヘッダーまたはチャット入力欄付近に **`?` ボタン** を1つ配置
- クリックで「このアプリでできること」を簡潔に表示
  - 例: 「自然文で運動を記録(『ベンチプレス60kg×8を3セット』など)」「種目別に記録をグラフ化」「怪我・目標などの文脈もメモ」
- 表示形式はモーダル or ツールチップ(影響範囲が小さい方を選ぶ)

設計判断の論点(実装前に決めること):
1. **配置場所**: ヘッダー右上 / 入力欄の隣 / その他
2. **表示形式**: モーダル / ツールチップ / アコーディオン
   - → モーダル推奨(モバイルでもタップしやすい)
3. **文言**: 機能リスト3〜5項目 + サンプル入力1〜2例

受け入れ条件:
- 画面上に `?` のような明確なヘルプ起点が見える
- クリックで「何ができるか」が3〜5行で分かる
- 閉じる操作で元のチャット画面に戻れる

面接で語れる設計判断:
- 「画面内チュートリアルは作らず、外部資料(スライド/README)に集約した」
  → 理由: MVP は触ってもらう人に最小コストで価値を伝えるべき。面接官など触らない層には別チャネル(スライド)で伝える方が情報密度が高い。

#### 要件B: エラー表示をユーザー向けにする

目的:
技術的な HTTP 500 などをそのまま見せず、ユーザーに分かる表現にする。

対応案:
- `エラー: HTTP 500` ではなく「一時的に応答に失敗しました。少し時間を置いてもう一度試してください。」と表示する

受け入れ条件:
- ユーザーに内部エラーコードを直接見せない
- 開発者側では console や backend log で原因を追える

#### 要件C: 前回の会話ログを復元する

目的:
再訪時に会話が消えたように見える違和感を減らす。

対応案:
- 既存の conversation history または chat_logs から最新ログを取得する API を追加
- フロント初期表示時に user_id ごとの直近会話を取得して messages に反映する

受け入れ条件:
- 同じユーザーで再ログインしたとき、直近の会話が表示される
- 少なくとも直近20件程度が見える

#### 要件D: 「記録がありません」表示を自然にする

目的:
現在選択している種目に記録がない場合の違和感を減らす。

対応案:
- 種目が存在しない場合は「まだ記録がありません」と汎用表示にする
- 種目選択がある場合のみ「○○の記録がありません」と表示する
- 初期種目を固定でベンチプレスにしない、または「すべて」表示を用意する

受け入れ条件:
- ランニング等のユーザーに「ベンチプレスの記録がありません」と出ない

## 7. 今回は将来課題として扱う要件

### 種目名訂正

重要だが、データ編集UI・更新API・誤抽出修正フローが必要なため、MVP提出後の改善候補とする。

### 定型入力フォーム

チャット入力が面倒な人向けに価値がある。  
ただし現在のMVPの主軸は「自然文から記録できる体験」なので、フォームは補助機能として後回し。

### 本格認証・セキュリティ

現在は友人テスト用の簡易認証。  
本番化する場合は Supabase Auth、RLS、Rate Limit、利用量制限、目的外利用対策を実装する。

## 8. 面接で説明するポイント

このMVPでは、まず価値検証を優先し、ニックネーム + 合言葉による簡易ユーザー分離を採用した。  
友人テストにより、チャット入力の自然さだけでなく、初期状態の分かりやすさ、会話ログの永続化、エラー表示、認証周りの課題が明確になった。

本番化するなら、Supabase Auth と RLS によりユーザーごとのデータアクセスを制御し、AI利用量制限や目的外利用対策を追加する。

## 9. 開発スタンス・学習方針

本プロジェクトの主目的は、単にLLMにMVPを作らせることではない。  
サマーインターンやメガベンチャーの面接で、実装・設計・技術選定について深掘りされた際に、自分の言葉で説明できる状態になることを重視する。

したがって、LLMは実装代行者ではなく、コーディングメンターとして利用する。

### 9.1 LLMに求める役割

LLMには以下を依頼する。

- スケジュール管理
- 実装順序の整理
- つまずいたときの原因切り分け
- どのファイル・関数を見るべきかの示唆
- 設計判断に対するフィードバック
- 面接で聞かれそうな深掘り質問の提示
- 調べるべき概念の提示
- コードレビュー
- プロンプト設計・Function Calling設計の相談
- セキュリティ・認証・デプロイ設計の論点整理

### 9.2 LLMに避けてほしいこと

LLMには以下を避けてほしい。

- 勝手にファイルを編集すること
- 勝手にコミット・pushすること
- 自分が理解しないまま大きな実装を進めること
- 設計判断の理由を説明せずにコードだけ提示すること
- MVPの目的から外れた過剰実装
- 面接で自分が説明できない抽象化やライブラリ導入

### 9.3 基本の進め方

各タスクは以下の流れで進める。

1. 今の目的を明確にする
2. 関係するファイル・関数を自分で予想する
3. なぜそのファイル・関数を変更する必要があるのか説明する
4. LLMからフィードバックを受ける
5. 自分でコードを書く
6. エラーが出たら、まず仮説を立ててからLLMに相談する
7. 実装後に「面接用30秒説明」を作る
8. 必要なら深掘り質問と回答を整理する

### 9.4 LLMへの質問形式

望ましい質問例:

- 今の目的は〇〇です。この場合、どのファイルのどの関数を見るべきですか？
- 私は〇〇を変更すべきだと思います。この考えは合っていますか？
- このエラーは〇〇が原因だと思うのですが、切り分け方を教えてください。
- この設計判断を面接で聞かれたら、どう説明すべきですか？
- この実装でセキュリティ上突っ込まれそうな点はどこですか？
- Function Calling と通常のJSON出力の違いを、この実装に即して説明したいです。

避けたい質問例:

- 全部実装して
- いい感じに直して
- 何でもいいから作って
- 面接で聞かれそうなことを全部まとめて

### 9.5 自分が説明できるべき主要論点

#### Webアプリ構成

- Vite / React は何をしているか
- FastAPI は何をしているか
- Uvicorn は何をしているか
- Vercel と Render の役割分担
- Supabase の役割
- localhost とローカルIPの違い
- CORS がなぜ必要か

#### データ設計

- workouts と context_notes と chat_logs を分けた理由
- user_id をどう生成しているか
- localStorage に何を保存しているか
- 本番化するなら Supabase Auth / RLS が必要な理由
- created_at が UTC 表示になる理由

#### LLM設計

- Function Calling を使った理由
- Structured Output との違い
- LLMが雑談まで保存しすぎる問題
- prompt で制御できることと限界
- 目的外利用・プロンプトインジェクションへの懸念
- RAG を本来どう使いたかったか

#### UX設計

- 友人テストで得た課題
- 初期状態で何ができるか伝える必要性
- エラーコードをそのまま見せない理由
- チャット入力とフォーム入力の使い分け
- 会話ログ復元が必要な理由

#### セキュリティ・運用

- 現在の簡易認証の限界
- API利用量制限の必要性
- LLMコスト管理
- 個人情報入力への注意喚起
- 友人テストと本番運用の違い
