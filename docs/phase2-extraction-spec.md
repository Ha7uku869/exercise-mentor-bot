# Phase 2: LLM抽出テスト・評価基盤 — 判断仕様

更新日: 2026-06-09

## 目的

経路A（Web `/api/chat`）の Function Calling が、ユーザー発話に対して
「どのツールを呼ぶべきか」を正しく判断できているかをテストするための、
**期待値（正解ラベル）の定義**。テストケースの「正解」はこの仕様を根拠にする。

## 対象と判断の3分類

対象: `src/api/main.py` の `chat_with_tools` + `TOOLS`
（`_build_chat_system_prompt` のルールに従って判断される）

判断は「保存する / しない」の2択ではなく **3分類**:

1. `save_workout` … 運動を「実施した」報告（過去・完了）
2. `save_context_note` … 将来の提案を変える**持続的**な情報（怪我 / 目標 / 生活・環境・習慣の変化）
3. なし(text) … 質問・相談・一時的なつぶやき・無関係な話

注意: 経路Aには**栄養専用ツールが無い**（`save_workout` と `save_context_note` のみ）。
栄養・サプリの習慣変化は `save_context_note`(category=lifestyle) に寄せる。

## 判断表

| 入力例 | 期待アクション | 理由（境界の根拠） |
|---|---|---|
| 今日ベンチ60kgを8回3セットやった | save_workout | 「実施した」過去の実績だから |
| 右肩を痛めたので来週まで休む | save_context_note | 将来の種目提案を変える情報だから |
| ベンチプレスってどう伸ばす? | なし(text) | 質問であって実績でも文脈でもない |
| 今日は疲れた | なし(text) | 一時的な気分で将来提案を変えない |
| 来週からジムに通い始める | save_context_note | 今後使える器具・環境が変わるから |
| 最近よく眠れてて調子いい | save_context_note ⚠️既知の失敗 | 睡眠は提案量に影響する（論点1: 方針(b)採用＝意図的に失敗させREADMEに記録） |
| 腰が痛い気がするけど多分大丈夫 | save_context_note | 曖昧でも怪我は安全側に倒して記録する |
| ベンチ60kg挙がるようになりたい | save_context_note (goal) | 明確な数値目標の宣言だから |
| 昨日5km走ったよ | save_workout | 「今日」でなくても完了した運動実績だから |
| プロテイン飲み始めた | save_context_note (lifestyle) | 摂取量・タイミングの提案につながる習慣変化だから |

## 境界の定義（ドラフト）

- **save_workout**: ユーザーが*実際にやった*運動の報告（過去・完了）。日付の明示が無ければ今日。
- **save_context_note**: *持続し、将来の提案を変える*事実。怪我 / 目標宣言 / 生活・環境・習慣の構造変化。
- **なし(text)**: 上記以外。アドバイスを求める質問、一時的な気分・体調のつぶやき、無関係な話。

> 境界を引く軸は「質問かどうか」ではなく **「持続性・将来の提案への影響があるか」**。
> 一時的な気分（疲れた・眠い・今日は調子いい）は、運動に関係していても保存しない。

## ⚠️ テスト前に決めるべき論点（未確定）

### 論点1: 「調子いい/睡眠」をどう扱うか（システムプロンプトとの不一致）

> **決定（2026-06-09）: 方針(b) を採用。** 期待値は `save_context_note` のまま残し、
> テストを意図的に失敗させ、READMEに「既知の失敗ケース＝プロンプト改善の宿題」として記録する。

- 現状のシステムプロンプト（`main.py` のツール使用ルール）は
  **「調子がいい」「眠い」を明示的に非該当(雑談)**とし「迷ったら呼ばない」と指示している。
- そのため「最近よく眠れてて調子いい」を `save_context_note` 期待にすると、
  LLMはプロンプト通り text（保存しない）を返し、**テストは失敗する可能性が高い**。
- これは「LLMのバグ」ではなく **「自分の意図」と「プロンプトの指示」がズレている**という発見。
- 対応の選択肢:
  - (a) 期待値を `なし(text)` に合わせる（プロンプトに従う）
  - (b) 期待値を `save_context_note` のまま残し、**「既知の失敗ケース」**としてREADMEに記録、
    後でプロンプトを改善する宿題にする（要件定義の「失敗ケースをREADMEに記録」に合致）
- 補足: 「今日よく眠れた」(一時的) と「最近ずっと寝不足」(慢性) は別物。
  慢性の睡眠不足の方が context_note 向き。

### 論点2: 「雑談」の定義が、表と矛盾している

- 当初の一般ルール案:「雑談 = アドバイスを求める質問 or 無関係な内容のみ」。
- このルールだと「今日は疲れた」は質問でも無関係でもない → 保存対象になり、
  判断表4行目（なし(text)）と**矛盾する**。
- 採用する境界（上の「境界の定義」）で解消する:
  雑談かどうかは「質問か」ではなく「**持続性・将来への影響があるか**」で判定する。

### 残課題（テスト設計上）
- 1発話で workout と note の両方を呼ぶべきケース（例: 「肩痛いけど今日は脚だけやった」）は
  まだ入れていない。複数tool_callの検証は基盤ができてから追加する。
- 「2回記録されてしまう」二重記録バグ（友人FB）は、抽出の正しさとは別問題。Phase 2の範囲外。

## 実行結果（2026-06-09）

`python -m pytest tests/test_extraction.py -v` → **25 passed, 1 xfailed (149s)**

- 純粋ユニットテスト2件（`_decision`）: 即緑。
- LLM判定 23件（W1-W8 / C1-C9 / N1-N6）: **全件正解 = 正答率100%**。
- X1: 設計通り **XFAIL**（現プロンプトは睡眠/体調を拾わない）。
- 補足: 初回はC6・N2が `RateLimitError(429)`（Groq無料枠のTPM=12000超過）で落ちた。
  分類ミスではなくインフラ制約。`_run` に `asyncio.sleep(5)` のthrottleを入れて解消。

### 再現性の担保（設計質問#5の答え）
1. `temperature=0`（`chat_with_tools`）で判定の揺れを抑える。
2. 純粋ロジック(`_decision`)は決定的テストに分離。
3. LLM判定は正答率で評価し、既知の失敗は xfail で明示（二値の合否ゲートにしない）。
