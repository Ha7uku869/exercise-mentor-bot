# 就活MVP 要件定義 2026-05-29

## 1. 目的

このMVPの目的は、就活のES・面接で「LLMアプリを作った」ではなく、以下を自分の言葉で説明できる状態を作ることである。

- ユーザーの自然文を構造化データに変換した経験
- 会話履歴や個人文脈を使って、次の提案を変えるAIを設計した経験
- LLMの出力をそのまま信じず、テスト・評価・安全性を考えた経験
- フロントエンド、バックエンド、DB、LLM APIをつないだ全体設計の経験
- 研究テーマである「対話ログの構造化」「ナレッジグラフ」「自己理解支援」へ自然に接続できる実装経験

対象企業は、トヨタ、NRI、LINEヤフー、NTTドコモを中心とする。

## 2. 方針

現MVPの題材は「運動メンター」のまま維持する。

ただし、見せ方は単なる運動記録アプリではなく、次の方向へ拡張する。

> 自然文の運動記録や相談をもとに、ユーザーの目標・怪我・生活変化・過去の会話を記憶し、根拠を持って次の行動を提案するPersonal AI Agent

研究テーマそのものに寄せすぎると、研究の未完成版に見えるリスクがある。一方で、研究と完全に無関係な技術スタックを増やすと、面接で「なぜその技術が必要だったのか」を説明しにくい。

そのため、本MVPでは以下のバランスを取る。

- 題材: 運動・筋トレ・健康習慣という身近なプロダクト
- 技術: LLM、Function Calling、RAG、構造化データ、評価、React/FastAPI/Supabase
- 研究接続: ユーザー文脈グラフ、因果関係、自己理解支援、安全な対話設計

## 3. 初学者向けの方向性説明

このMVPは、最初は「チャットで運動記録を入力すると、AIが内容を読み取ってDBに保存し、グラフで振り返れるアプリ」である。

例:

```text
ユーザー: 今日ベンチプレス60kgを8回3セットやった
AI: ベンチプレス 60kg 8回 3セットとして記録しました
```

ここまでは、AIが自然文を表形式のデータに変換するアプリである。

次に作るのは、AIが「記録するだけ」ではなく、ユーザーの文脈を覚えて提案を変える仕組みである。

例:

```text
ユーザー: 最近右肩が痛い
AI: 右肩の痛みを記憶しました。しばらく高重量のプレス系は控えめにしましょう。

数日後:
ユーザー: 今日何やればいい？
AI: 右肩の痛みがあるので、今日は脚か有酸素を中心にするのがよさそうです。
```

この時、AIが使う情報は大きく3種類ある。

1. 今回の発言
2. 過去の会話や記録
3. 怪我・目標・生活変化などの長期的な文脈

さらに発展させると、これらをグラフとして表現できる。

```text
右肩の痛み -> プレス系種目を制限
減量目標 -> 有酸素提案を増やす
睡眠不足 -> 高強度トレーニングを避ける
```

これが「ユーザー文脈グラフ」である。研究で扱うナレッジグラフや因果ループの考え方に近いが、就活MVPでは難しく作りすぎず、ユーザーに見える価値を優先する。

## 4. 面接で伝える完成イメージ

最終的には、次のように説明できる状態を目指す。

> 自然文で入力された運動記録や相談をLLMで構造化し、Supabaseに保存するWebアプリを開発しました。単に記録するだけでなく、怪我や目標などの長期的な文脈を記憶し、次回以降の提案に反映する設計にしています。また、LLMの抽出結果が常に正しいとは限らないため、保存すべきケースと保存すべきでないケースのテストデータを作り、抽出精度や失敗パターンを確認できるようにしました。今後はユーザー文脈をグラフとして可視化し、なぜその提案になったのかを説明できるAIエージェントに発展させたいと考えています。

## 5. 開発ルール

このプロジェクトでは、LLMを実装代行者ではなく、有能なメンターとして使う。

目的は、短時間で完成品を作ることではない。面接で深掘りされたときに、自分の言葉で設計・実装・失敗・改善を説明できる状態になることである。

### 5.1 LLMに任せてよいこと

- 要件整理
- 実装順序の提案
- 関係ファイルの特定
- エラー原因の切り分け
- 設計判断の壁打ち
- コードレビュー
- 面接で聞かれそうな質問の作成
- READMEや説明資料の下書き

### 5.2 LLMに任せすぎないこと

- 何を作るかの最終判断
- 重要なコードを理解しないまま貼り付けること
- 大きな実装を一気に進めること
- なぜその技術を使ったかを説明できないまま導入すること
- テストせずに「動くはず」と判断すること

### 5.3 各タスクの進め方

各実装タスクは、必ず以下の流れで進める。

1. 今回の目的を1文で書く
2. 変更するファイルを予想する
3. なぜそのファイルを変更するのか説明する
4. LLMに方針レビューを依頼する
5. 自分で小さくコードを書く
6. 動作確認する
7. エラーが出たら、まず自分の仮説を書く
8. LLMに切り分けを相談する
9. 実装後に30秒説明を作る
10. 面接深掘り質問を3つ作る

## 6. 60時間スコープ

### Phase 0: 要件固定と現状把握 3時間

目的:
迷走を防ぎ、どこまで作れば就活上十分かを決める。

やること:

- 現MVPの機能を整理する
- 現在の技術スタックを一覧化する
- 60時間の到達目標を決める
- READMEに書くべき論点をメモする

完了条件:

- この要件定義を読み、優先順位に合意している
- 最初に着手する実装タスクが決まっている

### Phase 1: 会話ログ復元 7時間

目的:
友人フィードバックで出ていた「再訪時に会話が消える」問題を解消する。

要件:

- `chat_logs` または既存の会話保存先から、ユーザーごとの直近会話を取得できる
- フロント初期表示時に、直近20件程度の会話を表示する
- ログインユーザーごとに会話が分離される

技術候補:

- FastAPI GET API
- Supabase `chat_logs`
- React `useEffect`

面接で語ること:

- ユーザーフィードバックをもとに改善した
- 会話型UIでは、過去文脈の復元が体験品質に直結する
- 本番化では認証とRLSが必要

### Phase 2: LLM抽出テスト・評価基盤 10時間

目的:
LLMを使った自然文抽出がどの程度正しいかを、自分で検証できる状態にする。

要件:

- 20〜40件のテストケースを作る
- 保存すべき発話、保存すべきでない発話を混ぜる
- 筋トレ、ランニング、ヨガ、怪我、目標、雑談を含める
- `pytest` で主要ケースを確認できる
- 失敗ケースをREADMEに記録する

例:

```text
保存すべき:
今日ベンチ60kgを8回3セットやった
30分ランニングして5km走った
右肩を痛めたので来週まで休む

保存すべきでない:
ベンチプレスってどうやれば伸びる？
今日は疲れた
プロテインって必要？
```

面接で語ること:

- LLMは便利だが、常に正しいとは限らない
- テストケースを作り、想定通り保存されるか検証した
- プロンプトだけでなく、評価データを持つことが重要だと学んだ

### Phase 3: 記憶カードUI 10時間

目的:
怪我・目標・生活変化など、AIが長期的に覚えている情報をユーザーが確認できるようにする。

要件:

- `context_notes` を一覧表示する
- categoryごとに表示する
  - injury
  - goal
  - lifestyle
  - other
- いつ保存されたかを表示する
- 有効期限がある場合は表示する
- ユーザーが「AIが何を覚えているか」を確認できる

面接で語ること:

- AIが記憶を使うなら、ユーザーが記憶内容を確認できるべき
- 透明性と信頼性を高めるために、記憶カードUIを追加した
- ドコモ・トヨタ向けには、人の状態理解と行動支援として説明できる

### Phase 4: ユーザー文脈グラフ 13時間

目的:
ユーザーの目標・怪我・行動・提案の関係をグラフとして扱う。

要件:

- ノードとエッジの最小データモデルを作る
- `context_notes` や `workouts` からグラフ要素を生成する
- React Flow または Cytoscape.js で簡単に可視化する
- 「なぜこの提案になったのか」を説明する材料にする

最小モデル:

```text
Node:
- id
- user_id
- type: goal / injury / workout / recommendation / lifestyle
- label
- source_id
- created_at

Edge:
- id
- user_id
- source_node_id
- target_node_id
- relation: affects / restricts / supports / caused_by / related_to
- created_at
```

例:

```text
右肩の痛み --restricts--> ベンチプレス
減量目標 --supports--> ランニング
睡眠不足 --affects--> 高強度トレーニング
```

研究室APIを使う場合:

- 研究室APIを本体ロジックに強く依存させない
- まずはMVP内の単純なグラフモデルを作る
- その後、研究室APIを「グラフ生成・関係抽出・可視化の補助」として接続する
- 面接では「研究室で学んだナレッジグラフの考え方を、個人開発の題材で小さく検証した」と説明する

面接で語ること:

- 研究テーマと個人開発を接続できる
- ただのRAGではなく、ユーザー文脈を構造として扱った
- なぜその提案になったかを説明しやすくした

### Phase 5: 軽量Multi-Agent化 9時間

目的:
1つの巨大なプロンプトに頼らず、役割を分けてAI処理を設計する。

要件:

- 以下の役割をコード上で分ける
  - Extractor: 自然文から構造化データを抽出する
  - Retriever: 過去の記録・文脈を取得する
  - Coach: ユーザーへの返答を作る
  - Safety: 危険な相談や目的外利用を判定する
- 最初は外部ライブラリを使わず、関数やモジュール分割で実装する
- 必要性が出た場合のみ、LangGraphなどを検討する

面接で語ること:

- Multi-Agentを流行語として使ったのではなく、責務分離として設計した
- LLM処理を抽出、検索、応答、安全性に分けることで、テストしやすくした
- LINEヤフーのAIエージェント系ポジションに接続しやすい

### Phase 6: 安全性・運用・説明資料 8時間

目的:
面接で深掘りされても崩れない説明を用意する。

要件:

- READMEを更新する
- 構成図を作る
- 面接30秒説明を作る
- 面接2分説明を作る
- 想定質問と回答を作る
- 現在の限界と本番化するなら必要な改善を書く

必ず説明する限界:

- 現在の認証は簡易的であり、本番ではSupabase AuthとRLSが必要
- LLM出力は誤るため、評価と人間による確認が必要
- 健康・メンタルヘルス領域では、医療助言にならないよう安全設計が必要
- API利用量制限、ログ管理、個人情報保護が必要

## 8. 企業別の見せ方

### トヨタ

刺す観点:

- 人の状態理解
- 安全・信頼できるAI
- 実世界の制約を考えた提案
- ユーザー文脈に応じた行動支援

説明:

> 運動という身近な題材で、ユーザーの状態や制約を記憶し、それに応じて提案を変えるAIを作りました。車載AIやHMIでも、人の状態や文脈を理解して安全に支援する設計が重要だと考えています。

### NRI

刺す観点:

- 課題整理
- プロトタイプ開発
- クラウド上で動くシステム
- 顧客課題をシステムに落とし込む力

説明:

> 友人フィードバックから課題を整理し、会話ログ復元やエラー表示、記憶UIなどの改善要件に落とし込みました。LLM活用だけでなく、DB設計、API設計、評価、運用課題まで含めてプロトタイプを改善しました。

### LINEヤフー

刺す観点:

- AIエージェント
- 大規模サービスを意識したバックエンド
- LLMの評価
- Python/TypeScriptでの実装力

説明:

> 自然文を構造化し、過去文脈を検索し、応答を生成する処理を役割ごとに分けて設計しました。LLMを単なるチャットとして使うのではなく、構造化・検索・評価・安全性を含むAIエージェントとして扱った点を工夫しました。

### NTTドコモ

刺す観点:

- ユーザー接点
- パーソナライズ
- データ活用
- AIエージェント
- プライバシーと透明性

説明:

> ユーザーの目標や怪我などの個人文脈を記憶し、次の提案に反映するAIを作りました。同時に、AIが何を覚えているかをユーザーが確認できる記憶カードUIを用意し、透明性を高める設計にしました。

## 9. 研究室APIの扱い

研究室APIを使える場合は、積極的に検討する。ただし、就活MVPとしては以下の方針を守る。

- APIがなくてもMVPの主要機能は動くようにする
- APIは研究接続を強める拡張機能として扱う
- APIの中身を理解せずに使ったとは言わない
- 何を入力し、何が返り、MVPのどの判断に使っているかを説明できるようにする

候補:

- 発話からイベント・感情・行動を抽出する
- 発話間の関係を抽出する
- `causedBy` / `similarTo` / `hasContext` のような関係を推定する
- ユーザー文脈グラフのノード・エッジ生成に使う

面接での言い方:

> 研究室で扱うナレッジグラフの考え方を、個人開発のMVPに小さく取り入れました。ただし、研究そのものを無理にプロダクト化するのではなく、まずユーザーに見える価値として「AIが何を根拠に提案しているか」を説明できるようにすることを優先しました。

最初にやる実装タスクは、会話ログ復元である。

着手前に自分で答えること:

1. 会話ログは今どこに保存されているか
2. フロントはどのタイミングでログを読み込むべきか
3. APIはどのURL・どのレスポンス形式にするべきか
